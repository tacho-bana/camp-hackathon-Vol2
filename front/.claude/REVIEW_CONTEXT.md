# REVIEW_CONTEXT

このファイルは Claude Code のレビュワーが読む、プロジェクト固有の方針メモ。

## プロジェクト概要

位置情報連動型タワーディフェンスゲームのフロントエンド。現実の街を歩いて店舗にチェックインし防衛拠点を設置、サーバー側で処理されるウイルス敵の波を守るゲーム。1週間ハッカソン・MVP段階。

## 採用している技術スタック

- フレームワーク: React 19 + Vite 8
- 言語: TypeScript 6.x
- 地図: Mapbox GL JS 3.x
- 状態管理: React Context API (`AppStateContext`)
- ルーティング: カスタム実装 (`window.history` + `popstate`)
- UIライブラリ: なし（CSS カスタム）
- バックエンド: FastAPI（別チーム担当、REST API で連携）

## このプロジェクト固有のルール

- `back/` ディレクトリは絶対に編集しない（読むのはOK）
- ゲームルール判定はすべてバックエンド側で行う（サーバー権威型）。フロントは表示と操作入力のみ
- MVPでは認証・ボス・音・演出・PvP・協力プレイは実装しない
- MVPでは店固有の能力は不要（全拠点を同じ能力に統一）
- 開発中は位置偽装モード（地図クリックで現在地を変える）を使うため、実機テスト不要
- バックエンド API エンドポイント: `POST /game/base`, `POST /game/start`, `POST /game/clear`, `POST /game/end`, `POST /structures/`, `GET /structures/`, `GET /enemies/`

## レビューで特に重視してほしい観点

- **動作の正確性**: ゲームパラメータ（50m判定・150コスト・30回収・家HP100・拠点HP100等）が仕様通りか
- **位置情報の扱い**: Geolocation API のエラーハンドリング（権限拒否・タイムアウト）が適切か
- **型安全性**: `any` を使わず、`LatLng` や `GamePhase` 等の型が正しく使われているか
- **API 呼び出し**: fetch のエラーハンドリング（network error, 4xx, 5xx）が適切か
- **状態の整合性**: `gamePhase` の遷移（waiting → prep → battle → result）が正しく管理されているか

## 無視してよい観点

- テストカバレッジ（ハッカソン期間中はテストを書かない）
- パフォーマンス最適化（MVP段階では不要）
- アクセシビリティ（今回のスコープ外）
- 認証・ログイン機能（MVPでは一旦無視）
- デザインの細かいピクセル調整（既存UIに合わせる方針）
